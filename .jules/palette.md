## 2024-03-08 - Keyboard accessibility and focus indicators for OS UI elements
**Learning:** Window close controls and start buttons lacking `:focus-visible` styles prevent effective keyboard navigation since the user cannot tell when these elements are focused using the Tab key. In OS UIs where contrast is low, `outline: 2px solid white` provides highly visible feedback. Additionally, icon-only window buttons lack screen reader / tooltip explanations of their functionality.
**Action:** Always ensure that dynamically created UI elements like window controls and custom buttons have both ARIA labels and `:focus-visible` styles, and utilize CSS-only tooltips for visual feedback.

## 2024-03-09 - Focus Trapping & Redundant Screen Reader Text in Dynamic UIs
**Learning:** In dynamically generated UI environments (like custom OS windows), failing to programmatically move focus to a newly opened window leaves keyboard users stranded on the trigger element (e.g., the desktop icon), effectively breaking tab navigation and rendering the window inaccessible. Furthermore, composite icon+text buttons (where the parent container is focusable) will cause screen readers to read the icon's `alt` text and the label redundantly (e.g. "Calculator, Calculator button") unless the decorative icon is explicitly hidden. Providing an OS-standard keyboard shortcut (Escape) to close active dialogs is also highly expected.
**Action:** When creating new interactive elements like modal windows via DOM manipulation, explicitly set `.focus()` on an internal control (like the close button) immediately upon creation, and map the `Escape` key globally to safely remove the highest `z-index` item. In composite focusable containers, explicitly set `alt=""` and `aria-hidden="true"` on inner presentation images to prevent redundant dictation.
